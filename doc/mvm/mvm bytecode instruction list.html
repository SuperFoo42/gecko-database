Mnemonic Opcode (Hex, 1 byte) Operand (8 byte) Stack ([top,...] → [top,...]) Description
tcreate 0E force [scope, pointer to string ] → table handle Creates a new table within the given scope with name as given by the string pointer. Leads to an ABORT if a table exists with the given name and force is not set to 1. Finishes the program with exit code 2 (ABORT BY SYSTEM), and a rollback is initialized. In case, force is set to 1 the table is dropped if it exists, and a new table with the given name is added. If successfully, an table handle to access the new table is pushed onto the stack. Note: the instruction tinstall must be placed somewhere below this instruction to complete the operation.
tinstall 13 [table type, table handle] → Installs the table given by table handle as a type of table type. Removes the table type argument from the stack.
tdrop 0F table handle → Drops the table given by table handle, and removes the table handle from the stack. In case no table with the given handle is found, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
dclone 04 pointer to string → Clones the current database into a path indicated by a string referenced by pointer to string. In case an I/O error occurs, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
dopen 05 force [timeout, pointer to string] → Opens the database stored in a path indicated by a string referenced by pointer to string. In case an I/O error occurs, an ABORT is invoked and a rollback is initialized, i.e., the current database remains open. The program exits with exit code 2 (ABORT BY SYSTEM) in this case. Note: Once this instruction is invoked, newly added transaction are automatically aborted until the opening process finishes. The parameter force defines the handling of running transactions. If force is set to 1, all running transactions are aborted immediately. If force is set to 0, this instructions waits until other running transactions are completed. If the waiting duration exceeds timeout milliseconds, other running transactions are aborted anyway and the opening process is issued.
tdup 11 [table handle (original), pointer to string] → table handle (copy) Constructs a copy of the table table handle (original) with the name referenced by pointer to string, and pushes the table handle onto the stack. In case no table with the id table handle (original) is found or there is another table with the name referenced by pointer to string, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
byname 02 pointer to string → table handle Finds the table handle associated with the name as given by pointer to string, and pushes table handle onto the stack. In case no table with the given name is found, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
tlist 14 → [num, table handle 0, table handle 1, ..., table handle (num-1)] Pushes all tables handles onto the stack. The element on top is a number indicating the number of installed tables.
tinsert 12 [num, table handle] → field handle Inserts num (uninitialized) tuplets into the table identified by table handle. Both input arguments are removed from the stack, an field handle is pushed onto the stack. The field handle points to the first field of the first inserted tuplet. In case num is less than or equal to zero, or the table handle is unknown, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
fwrite 07 [ data, field handle ] → field handle Writes data into the field associated with field handle, moves field handle to the next field and removes data from the stack. Which data type is used for data is inferred by the associated column of the table. In case the field handle is moves beyond the last field of the current tuplet, it is set to the first field in the subsequent tuplet. In case the field handle is moved outside the bounds of the tbale, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
proglist 17 → [num, prog id 0, prog id 1, ..., prog id (num-1)] Pushes all program ids onto the stack. The element on top is a number indicating the number of installed programs.
proginfo 16 prog id → Prints information to stdout about the program associated with prog id.
progclean 15 → Removes uninstalled programs from the program pool.
sleep 0C time → Suspends the thread for this transaction for time milliseconds.
tinfo 11 table handle → Displays information to stdout about the table associated with the handle table handle.
rjmp_nz 06 pos Continues the program execution at line pc + pos if the variable VARIABLE_RCX is not-zero. Therefore, this instruction implements a relative jump in the program flow w.r.t. the current line number of the rjmp_nz instruction. Note: the value of VARIABLE_RCX is interpreted as signed 8-byte integer.
addcol 01 [table handle, pointer to string, type, rep, flags ] → [column handle, table handle] Adds a new column with the name references by pointer to string to the table associated with table handle. The data type is defined by type. For fixed-length field types (e.g., VARCHAR(n)) set rep to n. The value flags is an xor-ed number defining the column flags. In case table handle is unknown, this action leads to an ABORT and the program finishes with exit code 2 (ABORT BY SYSTEM). If successfully, an column handle to access the new column and the original table handle are pushed onto the stack
vstore 19 i Stores the top value of the stack into an u64 variable with index i. The stack is not modified.
vload 18 i → value Loads the value of the variable with index i and pushes the value onto the stack.
isset 09 i → bool Checks whether the variable with index i was set before by vstore or similar. Pushes 1 onto the stack if so otherwise 0.
pop 0B value → Removes the top value from the stack
npop 0A [num, value 0, value 1,..., value (num-1)] → Removes num values from the stack
push 08 value → value Pushes value onto the stack as an u64 value
swap 0D [value1, value2] → [value2, value1] Swaps the two top elements of the stack
commit 03 Finishes the program with exit code 0 (COMMIT). Changes to the system are permanent.
abort 00 Finishes the program with exit code 1 (ABORT BY USER). A rollback is initialized, and changes to the system are discarded.
jlist 1A job state → [num, job id 0, job id 1, ..., job id (num-1)] Pushes the job ids of jobs having a state defined job state onto the stack. The job state can cover mutiple state type by xor-ing the state values. The element on top is a number indicating the number of jobs.
jinfo 1B job id → Prints information to stdout about the job associated with job id.
jclean 1C → Removes all non-running jobs from the job pool.
jkill 1D job id → Kills a running job associated with job id. In case the job id is unknown or the job is not running, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
jsusp 1E job id → Pauses a running job associated with job id. In case the job id is unknown or the job is not running, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
jwake 1F job id → Continues the execution of a paused job associated with job id. In case the job id is unknown or the job is not paused, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
jstart 20 prog id → job id Starts the program associated with prog id as a new job job id. The control flow continues execution. In case the prog id is unknown, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case.
acq_lock 21 container [mode, element] → lock id Acquires a mode lock in container on a specific element. The definition of element depends on the container. The job-thread might be suspended until the lock is available rather than doing a busy-waiting as for latches. In case the mode, container or element is unknown, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case. Note: locks can be explicitly released via the rel_lock instruction. However, when the program finishes execution, all locks that are not manually released will be released automatically.
rel_lock 22 lock id → Releases a lock associated with lock id that was previously aquired via the acq_lock instruction. In case lock id is unknown or the lock associated with lock id is already released, the program continues without any interruption. Note: When the program finishes execution, all locks that are not manually released via rel_lock will be released automatically.
acq_latch 23 container [mode, element] → latch id Acquires a mode latch ('spin-lock') in container on a specific element. The definition of element depends on the container. The job-thread will spin in a while-loop until the latch is available rather than being suspended as for locking. In case the mode, container or element is unknown, an ABORT is invoked and a rollback is initialized. The program exits with exit code 2 (ABORT BY SYSTEM) in this case. Note: latches can be explicitly released via the rel_latch instruction. However, when the program finishes execution, all latches that are not manually released will be released automatically.
rel_latch 24 latch id → Releases a latch associated with latch id that was previously aquired via the acq_latch instruction. In case latch id is unknown or the latch associated with latch id is already released, the program continues without any interruption. Note: When the program finishes execution, all latches that are not manually released via rel_latch will be released automatically.
vmove 25 i value → Stores the top value of the stack into an u64 variable with index i and removes value from the stack.
dec 26 i Decrements the value of the variable with index i.
temp_name 27 → pointer to string Creates a new (temporary) string on heap and assigns an unique table name to it. Then, a pointer to that string is pushed onto the stack. When the program finishes, all strings created by this instruction will be automatically freed. Note: the temporary string is guaranteed to be unique in local scope. Therefore, it is not reasonable to use this instruction to generate gobal table names.

Column Flags

Flag Name Value Restrictions Description
PRIMARY 1 No combination with FOREIGN or NULLABLE Defines a column as primary key column. Multiple columns can have this flag. The flag UNIQUE is automatically turned on, if this flag is set.
FOREIGN 2 No combination with PRIMARY or NULLABLE Defines a column as foreign key column. Multiple columns can have this flag.
NULLABLE 4 No combination with PRIMARY, FOREIGN or AUTOINC Turns on null values for a column. Multiple columns can have this flag.
AUTOINC 8 No combination with NULLABLE or FOREIGN Newly added fields in this column are automatically incremented.
UNIQUE 16 Newly added fields must contain values that are unique in this column. Null values are treated as one single (unique) value. Hence, having this flag applied, only one null value is allowed.

Lock and Latch Modes

Name Type Id Description
MODE_SHARED 0x0000000000000000 A shared (read) lock/latch will be acquired. Having a shared lock/latch on a specific element inside an container will not block other shared locks/latches on the same element inside this container. However, if an other caller requests an exclusive lock/latch for an element for which a shared lock is hold, the caller will be blocked until the shared lock/latch is released. Locks and latches are equivalent regarding these modes when used in combination on the same element.
MODE_EXCLUSIVE 0x0000000000000001 An exclusive (write) lock/latch will be acquired. Having an exclusive lock/latch on a specific element inside an container will bock both other shared and exclusive locks/latches on the same element in that container. Callers for other locks/latches will be blocked until the shared lock/latch is released. Locks and latches are equivalent regarding these modes when used in combination on the same element.

System Containers

Container Name Container Id Element(s) Description
CONTAINER_PROGPOOL 0x0000000000000000 ACCESS_GLOBAL(0x0000000000000000) The program pool (progpool) is the repository for compiled programs inside a MondrianDB instance. The granularity for locks/latches on this container is global which means that the entire repository is exclusively accessed.

Table Scopes

Name Type Id Description
GLOBAL 0 The table is stored in the global space of the database, and is visible for all transactions in the entire system. Global tables continue their existence after the program in which the table was firstly created finishes. Since global tables are a shared resource, accessing global tables typically involve the acquisition of a latch or lock. Hence, accessing global tables is more heavyweight than accessing local tables.
LOCAL 1 The table is stored in the local space of the program, and is visible only for the transaction that created the table. Local tables automatically end their existence when the program in which the table was created finishes. Since local tables are a non-shared resource, accessing local tables does not require to acquire a latch or lock. Hence, accessing local tables is more lightweight than accessing global tables.
